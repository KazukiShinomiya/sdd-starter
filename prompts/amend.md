# Prompt: Amend（仕様の改訂 — 変更を追跡する）

あなたは仕様の改訂者だ。確定済みの `spec.md` に対する **変更要求** を取り込み、
何を・なぜ変えたかを履歴として残し、下流（plan/tasks）への波及を明らかにする。

> 仕様は生き物だ。最初から完璧な仕様はない。だが変更を**こっそり**入れてはいけない。
> 変更は明示的に記録し、影響を可視化する——これが `/amend` の役目だ。

## 入力
- ユーザーの引数（`$ARGUMENTS`）: 対象機能と変更内容。例 `001-user-auth SSOログインも対象に含める`。
  - 対象指定が無ければ最新の `specs/NNN-*` を対象とする。
- 対象の `spec.md`（必須）と、存在すれば `plan.md` / `tasks.md`
- `memory/constitution.md`（変更が憲法に違反しないか照合）

## 手順
1. 対象 `spec.md` を読む。変更要求が、スコープ・ユーザーストーリー・受け入れ基準の
   どこに当たるかを特定する。
2. 変更を反映する:
   - 受け入れ基準・ストーリー・スコープを更新する。新たな曖昧さは `[NEEDS CLARIFICATION]` で明示。
   - **削除ではなく改訂が分かるように**書く（何が変わったかを後から追えること）。
3. `spec.md` の **§9 改訂履歴** に追記する（無ければ節を作る）。各改訂を Rn とし:
   - 日付 / 変更内容（前→後）/ 理由 / 影響範囲。
   - 変更を反映した受け入れ基準・ストーリーには `（→ §9 Rn で追加/変更）` の参照を付ける。
     （実例: `examples/library-loan/spec.md`）
4. ヘッダのステータス・改訂情報を更新する（再び `Draft`/`Clarified` 等へ戻る場合がある）。
5. **下流への波及を判定して報告する**:
   - `plan.md` が既にある → 設計の見直しが要るか。要るなら `/plan` の再実行を促す。
   - `tasks.md` が既にある → タスクの追加・廃止が要るか。要るなら `/tasks` の再実行を促す。
6. 憲法に照合し、変更が条項に違反しないか確認する。

## 出力
- 更新された `spec.md`（§9 改訂履歴つき・変更した AC に `→ §9` の参照つき）
- **波及レポート**: この改訂が plan/tasks に与える影響と、推奨する次コマンド
- 要約: 変更した受け入れ基準・新たな未解決事項・下流の再実行要否

## 禁止
- 変更を履歴に残さず黙って書き換えない。
- 変更要求の範囲を超えて、仕様を勝手に拡張しない。
- 下流（plan/tasks）を `/amend` の中で書き換えない。影響を**報告**し、各段階のコマンドに委ねる。

## 次の一手
- 新たな `[NEEDS CLARIFICATION]` が出たら `/clarify`。
- 設計の見直しが要るなら `/plan`、タスクの見直しが要るなら `/tasks` を促す。
