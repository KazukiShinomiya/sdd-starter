# Prompt: Checklist（成果物の品質チェックリスト生成）

あなたは品質ゲートの設計者だ。指定された **1つの成果物**（spec / plan / tasks）が
「良い成果物の条件」を満たすかを、レビュー担当者が機械的に検証できる
**観点チェックリスト** に変換する。

> `/analyze` との違い: `/analyze` は spec・plan・tasks の **三者間の整合** を見る。
> `/checklist` は **1つの成果物の内部品質** を見る。役割が重ならないよう、横断検証はしない。

## 入力
- ユーザーの引数（`$ARGUMENTS`）: 対象機能と観点。例 `001-user-auth spec` / `plan`。
  - 対象成果物の指定が無ければ最新の `specs/NNN-*` の `spec.md` を対象とする。
  - 観点（focus）の指定が無ければ成果物の種別に応じた既定観点を使う。
- 対象の成果物ファイル（`spec.md` / `plan.md` / `tasks.md`）
- `memory/constitution.md`（品質の上位基準として）

## 手順
1. 対象成果物を読む。種別（spec/plan/tasks）を特定する。
2. その種別に応じた品質観点で、**そのまま検証できる Yes/No 項目** に落とす:
   - **spec**: 受け入れ基準はすべてテストに変換可能か / 測定不能な語（速い・使いやすい）が無いか /
     スコープの含む・含まないが明記されているか / 未決定が `[NEEDS CLARIFICATION]` で可視化されているか /
     実装の話が混入していないか。
   - **plan**: 各設計判断に「却下案と理由」があるか / Constitution Check が全条項を被覆するか /
     仕様に無い機能を足していないか / 早すぎる最適化・不要な抽象化が無いか / リスクと緩和策があるか。
   - **tasks**: 全タスクに「完了の定義」があるか / 依存関係が明示されているか /
     トレーサビリティ表が全ACを被覆するか / 粒度が30分〜半日に収まるか / 宙に浮いたタスクが無いか。
3. 各項目は **成果物のどこを見れば判定できるか** を添える（節番号・ID など）。
4. プロジェクト固有の懸念（憲法の条項など）があれば観点に足す。想像で水増ししない。

## 出力
- `specs/NNN-feature-name/checklists/<focus>.md`（例 `checklists/requirements.md`）。
  各項目は `- [ ]` のチェックボックスで、レビュー担当者がそのまま埋められる形にする。
- 要約: 対象成果物・観点・項目数・特に注意すべき項目。

## 禁止
- 成果物そのものを書き換えない（チェックリストを別ファイルに生成するだけ)。
- 製品の機能チェック（「ログインできるか」等）を書かない——それは spec の受け入れ基準の領分。
  ここで問うのは「成果物が良い成果物か」だ。
- 判定不能な観点（「十分か」等)を入れない。Yes/No で答えられる形にする。

## 次の一手
- チェックリストで不備が見つかったら、対応する段階（`/specify` `/clarify` `/plan` `/tasks`）に戻す。
- 問題が無ければ次段階に進めると伝える。
