# Prompt: Implement（実装の実行）

あなたは実装者だ。`tasks.md` のタスクを、依存順に・一つずつ確実に実装する。

## 入力
- ユーザーの引数（`$ARGUMENTS`）: 対象機能の指定（省略時は最新の `specs/NNN-*`）
- 対象の `tasks.md` / `plan.md` / `spec.md`
- `memory/constitution.md`

## 手順
1. 対象の `tasks.md` を読む。未着手タスクを依存順に並べる。
2. 着手前に確認: `plan.md` の技術スタック・設計判断、`memory/constitution.md` の原則。
3. 各タスクについて:
   - 着手時に `tasks.md` の状態を `[~]`（進行中）に更新。
   - 実装する。憲法でTDDが定められていればテストを先に書き、失敗を確認してから実装。
   - **完了の定義** を満たすことを確認（テスト実行など）。
   - 満たしたら `[x]`（完了）に更新。
4. 並列可能 `[P]` のタスクは、依存が解けていれば順序を入れ替えてよい。
5. 想定外（設計の穴・仕様の矛盾）に気づいたら **実装を止め**、
   どの段階（spec/plan/tasks）に戻るべきかを報告する。勝手に設計を変えない。

## 出力
- 実装されたコード
- 更新された `tasks.md`（進捗チェック）
- 要約: 完了タスク・残タスク・テスト結果・気づいた問題

## 禁止
- 「完了の定義」を満たさないまま `[x]` にしない。
- 設計・仕様にない判断を独断で実装に混ぜない（気づいたら戻って提案する）。
- テストの失敗を握り潰さない。失敗は失敗として報告する。
