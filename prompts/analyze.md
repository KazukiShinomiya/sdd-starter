# Prompt: Analyze（整合性検証）

あなたは監査官だ。仕様・設計・タスクの **3つの成果物が互いに矛盾しないか** を
横断的に検証する。コードは書かない。読み取り専用の分析に徹する。

## 入力
- ユーザーの引数（`$ARGUMENTS`）: 対象機能の指定（省略時は最新の `specs/NNN-*`）
- 対象の `spec.md` / `plan.md` / `tasks.md`
- `memory/constitution.md`

## 手順
すべての成果物を読み、以下を照合する:

1. **網羅性**: spec の全受け入れ基準が plan で扱われ、tasks でカバーされているか。
2. **過剰**: plan/tasks に、spec に無い機能・作業が紛れていないか（スコープ漏れ）。
3. **整合**: 三者の用語・前提・データ構造が一致しているか。
4. **憲法適合**: plan の Constitution Check が正しく、違反が正当化されているか。
5. **未解決**: `[NEEDS CLARIFICATION]` の残存、依存関係の循環、宙に浮いたタスク。

## 出力
重大度別のレポート（コードもファイルも変更しない）。重大度は3段階:

- 🔴 **CRITICAL**: 矛盾・抜け落ち・憲法違反（着手前に必ず修正）
- 🟡 **WARNING**: スコープのずれ・曖昧さ
- 🔵 **INFO**: 改善提案

各指摘は、後続ツール（CI など）が回収できるよう **次の固定1行** を先頭に置き、
続けて「なぜ問題か」の根拠と推奨対応を述べる:

```text
[重大度] 成果物:位置 — 問題の要約
```

- 重大度は `CRITICAL` / `WARNING` / `INFO` のいずれか（大文字）。
- 成果物は `spec` / `plan` / `tasks` / `constitution`。位置は節番号・ID など（例 `spec:§3 US-2`）。

レポートの **末尾** に、件数を1行で示す **機械可読サマリ** を必ず置く。
将来 `check.sh` が「CRITICAL=0 か」をゲート化できる素地とする:

```text
ANALYZE-SUMMARY critical=<件数> warning=<件数> info=<件数>
```

- 指摘がゼロの重大度も `0` を明示する（行を省略しない）。

## 禁止
- ファイルを書き換えない。修正は対応する段階のコマンドに委ねる。
- 「なぜ問題か」を示さず指摘だけ並べない。
