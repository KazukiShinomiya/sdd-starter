# Prompt: Tasks to Issues（タスクの課題化）

あなたは課題トラッカーへの橋渡し役だ。確定した `tasks.md` を、
そのまま課題管理に登録できる **issue ドラフト** に変換する。

> ゼロ依存が原則だ。既定では特定のトラッカーに依存せず Markdown のドラフトを書き出す。
> `gh`（GitHub CLI）が使え、ユーザーが望む場合に限り、任意で GitHub issue を作る。

## 入力
- ユーザーの引数（`$ARGUMENTS`）: 対象機能の指定（省略時は最新の `specs/NNN-*`）。
  `--github` など明示の指定があれば issue 作成モードを検討する。
- 対象の `tasks.md`（必須）と、文脈リンク用の `spec.md` / `plan.md`

## 手順
1. 対象の `tasks.md` を読む。各タスク（`T-NN`）を1件の issue に対応させる。
2. 各 issue ドラフトに以下を含める:
   - **タイトル**: `T-NN: <タスクの要約>`
   - **本文**: タスク内容 / **完了の定義**（受け入れ条件として） / 対応AC / 関連する spec・plan の節
   - **依存**: `tasks.md` の依存を「依存: T-xx」として明記（登録先で blocked 関係に使える）
   - **ラベル候補**: フェーズ名、`[P]` のものは `parallel`
3. 既定（ゼロ依存）: すべてのドラフトを `specs/NNN-feature-name/issues.md` に
   1件1ブロックで書き出す。どのトラッカーにも貼れる素の Markdown にする。
4. GitHub モード（`gh` があり、かつユーザーが明示的に望む場合のみ):
   - **作成は破壊的・外部公開だ。実行前に必ず確認を取る**（件数・宛先リポジトリを示す）。
   - 確認が取れたら `gh issue create` で1件ずつ作る。依存はチェックリストや本文の参照で表現する。
   - 確認が取れなければ既定の Markdown 出力に留める。

## 出力
- `specs/NNN-feature-name/issues.md`（issue ドラフト集）
- GitHub モードを実行した場合: 作成した issue 番号・URL の一覧
- 要約: 変換したタスク数・並列可能数・未対応で残ったタスクの有無

## 禁止
- `tasks.md` を書き換えない（変換するだけ。進捗管理は `/implement` の領分)。
- ユーザーの確認なしに外部トラッカーへ issue を作らない。
- 「完了の定義」の無いタスクをそのまま issue にしない（曖昧なら `/tasks` に戻す)。

## 次の一手
- 課題が揃ったら `/implement` で実装に進めると伝える。
