## 1. 技術スタック

| 領域 | 選択 | 理由 |
|------|------|------|
| 言語 / ランタイム | Python 3.12 | 標準ライブラリだけで完結でき、ゼロ依存を満たす |
| 主要フレームワーク | なし（標準ライブラリ `argparse`, `re`） | CLI に外部依存は過剰 |
| テスト | pytest | デファクト。純関数を単体テストしやすい |

> 非機能要件「ゼロ依存」を最優先。フレームワークを入れない判断そのものが設計だ。

## 2. アーキテクチャ概要

入力テキスト → 見出し抽出 → 目次生成 → （マーカーがあれば）差し込み、という一方向のパイプ。
各段を純関数にして、ファイル I/O から切り離す。

```
cli.py (argparse: ファイルパスを受ける)
  └─ toc.py
       ├─ extract_headings(text) -> [Heading]   … コードブロックを除外して見出しを拾う
       ├─ render_toc(headings) -> str            … 階層付きの Markdown リストを作る
       └─ splice(text, toc) -> str               … マーカー間を置換（無ければ原文のまま）
```

## 3. データモデル

> CLI のため極小。別ファイルに切り出すまでもない（plan に畳む）。

| 構造 | フィールド | 説明 |
|------|-----------|------|
| Heading | level(int 1-6), text(str), anchor(str) | 1つの見出し。anchor は GitHub 互換規則で生成 |

## 4. インターフェース / 契約

> 外部境界は CLI 1つ。`contracts/` に切り出すほどではないため、ここに記す。

```
toc <file.md>            # マーカーがあれば更新（上書き）、無ければ目次を stdout に出力
  終了コード: 0=成功 / 1=ファイル無し・読めない / 2=引数不正
```

## 5. 主要な設計判断

- **判断1**: 見出し抽出を純関数にする ← 却下案: ファイルを直接読みながら処理。
  理由: テスト容易性（憲法のテスト駆動）。文字列 in / 構造 out なら I/O 無しでテストできる。
- **判断2**: コードブロック除外を正規表現＋状態フラグで行う ← 却下案: 完全な Markdown パーサ導入。
  理由: 「単純さ優先」。要件は見出しの抽出だけで、フルパーサは過剰。

## 6. Constitution Check（憲法適合チェック）

> この例では「単純さ優先」「テスト駆動」「明示性」を前提に照合した。
