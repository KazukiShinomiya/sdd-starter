## 1. 技術スタック

- **言語 / 実行**: Python 3.12（標準ライブラリ中心、ゼロ依存に近い構成）
- **永続化**: SQLite（単一ファイル・組み込み。図書館1館規模の同時実行に十分）
- **配布形態**: 職員端末で動くローカルアプリ（CLI + 簡易 Web UI）
- **却下案**: フルRDB（PostgreSQL）→ 1館規模に運用コストが過剰。SQLite で要件を満たす。

## 2. アーキテクチャ概要

```
[職員/利用者操作] → アプリ層（貸出・返却・延長・予約のユースケース）
                         │
                  ドメイン層（期限計算・延滞金・待ち行列・確保期限のルール）
                         │
                  永続層（SQLite: 蔵書 / 貸出 / 予約）
```

- ルール（期限・延滞金・予約優先）は **ドメイン層に集約**し、UI とDBから独立させる。
- 「休館日」はカレンダーをドメイン層が持ち、期限・確保期限の計算に使う。

## 3. データモデル

| エンティティ | 主な属性 | 備考 |
|------------|---------|------|
| Book | id, title, copies_total | 蔵書（同一タイトルに複数 copy がありうる） |
| Loan | id, book_id, user_id, loaned_at, due_at, returned_at, extensions_used | 貸出記録。`extensions_used` で延長回数を追う |
| Reservation | id, book_id, user_id, queued_at, status, hold_until | **R1で追加**。待ち行列と確保期限 |

- `Reservation.status`: `waiting`（待機）/ `held`（確保中）/ `fulfilled`（貸出済）/ `expired`（流れた）。
- `hold_until`: 確保通知後に本を押さえる期限（**3開館日**）。`held` の間だけ有効。

## 4. インターフェース / 契約

- `borrow(book_id, user_id) -> Loan | Error` — 在庫と冊数上限を確認して貸出。
- `return_book(loan_id) -> ReturnResult` — 返却。延滞金を算定し、**予約待ち行列があれば先頭を `held` にして確保通知を出す**。
- `extend(loan_id) -> Loan | Error` — 期限内かつ予約が無い場合のみ延長（残り回数を確認）。
- `reserve(book_id, user_id) -> Reservation | Error` — **R1で追加**。在庫が無い本にのみ予約可。

## 5. 主要な設計判断

- **延滞金の上限はドメイン層で頭打ち**: `min(延滞日数 × 10, 500)`。UI/DB に散らさない（AC-2.2）。
- **期限計算は「暦日 + 休館日繰り延べ」**: 14日後を求め、休館日なら次の開館日へ送る（AC-1.2）。
- **R1（予約）の波及** — spec 改訂を受けた設計変更:
  - データモデルに `Reservation` を追加し、`return_book` の処理に **確保通知の分岐**を足した。
  - `extend` に **予約有無のチェック**を追加（予約者優先、AC-3.2）。
  - 確保期限 `hold_until` も期限計算と同じ「開館日」基準に揃えた。
  - → 予約機能は「あとから足した」が、ルールはドメイン層に閉じ込めたため UI/DB への波及は最小に留めた。
- **却下案**: 予約を `Loan` に相乗りさせる案 → 状態遷移（待機/確保/流れ）が貸出と異なり混線する。
  独立した `Reservation` に分けた。

## 6. Constitution Check（憲法適合チェック）

- [ ] **利用者の公平な扱い**: 予約は先着順の待ち行列、延長は予約者優先 → 公平性を具体化。適合。
- [ ] **データの一貫性**: 在庫の二重貸出を防ぐ（返却→確保→貸出の遷移を単一トランザクションで扱う）。適合。
- [ ] スコープ外（蔵書購入・利用者認証）には踏み込んでいない。適合。

## 7. リスクと未確定事項

- **確保期限の起算日**が未確定（spec §6 の `[NEEDS CLARIFICATION]`）。実装前に `/clarify` で確定する。
- **同時返却と予約確保の競合**: 同じ本の copy が複数あると確保のひもづけが複雑化。
  → 当面は「copy 単位ではなくタイトル単位で先頭予約者に確保」で簡素化し、リスクとして記録。
