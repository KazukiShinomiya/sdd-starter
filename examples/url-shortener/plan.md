## 1. 技術スタック

| 領域 | 選択 | 理由 |
|------|------|------|
| 言語 / ランタイム | Python 3.12 | 標準ライブラリが厚く、最小依存で書ける |
| 主要フレームワーク | FastAPI | 軽量・型に基づく検証・自動ドキュメント |
| データストア | SQLite | 単一プロセス・ゼロ運用。要件の規模に十分 |
| テスト | pytest | デファクト。FastAPI の TestClient と相性良 |

> 「指定がなければ最も単純で枯れた選択」の原則に従った。
> 規模が小さくスケール要件も緩いため、RDBや分散KVは過剰と判断。

## 2. アーキテクチャ概要

3層の薄い構成。ビジネスロジックを `core` に隔離し、Web層から独立させる。

```
HTTP (FastAPI ルータ)
  └─ core/shortener.py   … 識別子生成・登録・解決・計測（フレームワーク非依存）
        └─ storage.py    … SQLite への永続化（インターフェース越し）
```

## 3. データモデル

| エンティティ | 属性 | 関係 |
|--------------|------|------|
| Link | id(PK), code(unique, 7文字), original_url, created_at, hits(int, default 0) | — |

- `code` に一意制約。発行時に衝突したら再生成（リトライ）。

## 4. インターフェース / 契約

| メソッド | パス | 役割 | 対応AC |
|----------|------|------|--------|
| POST | `/links` | URLを登録し `{code, short_url}` を返す | AC-1.1〜1.3 |
| GET | `/{code}` | 元URLへ302リダイレクト。無ければ404 | AC-2.1, 2.2 |
| GET | `/links/{code}/stats` | `{code, hits}` を返す | AC-3.1, 3.2 |

## 5. 主要な設計判断

- **判断1**: 識別子は乱数Base62×7文字 ← 却下案: 連番のBase62エンコード。
  理由: 連番は総数や登録順を推測可能にする。乱数で推測を防ぐ。
- **判断2**: `core` をフレームワーク非依存に隔離 ← 却下案: ルータに直書き。
  理由: 「単純さ優先」と相反するようだが、テスト容易性（憲法のTDD）を満たすため
  最小限の層分けは正当。Web起動なしにロジックをテストできる。

## 7. リスクと未確定事項

- **リスク**: 識別子の衝突。→ 一意制約＋リトライで緩和。確率は7文字Base62で極小。
